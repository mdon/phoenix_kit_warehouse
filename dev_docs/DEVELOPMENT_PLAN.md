# PhoenixKit Warehouse — план доработки

Дата: 2026-07-10. Источник идей: анализ Цеховик ERP (cehovik-erp.ru, см. разбор в `phoenix_kit_manufacturing/dev_docs/DEVELOPMENT_PLAN.md` §А) + подготовка к запуску производственного модуля. Связанные планы: locations (`phoenix_kit_locations/dev_docs/DEVELOPMENT_PLAN.md`), производство (`phoenix_kit_manufacturing/dev_docs/DEVELOPMENT_PLAN.md`).

## 0. Что уже есть

Документы: приёмки (goods_receipts), расходы (goods_issues), внутренние заказы (internal_orders), заказы поставщикам (supplier_orders), инвентаризации (inventory_documents). Остатки: `stock` + stock_ledger + committed_quantities. Инфраструктура: activity log, комментарии, файлы (storage_folders), view_configs/column_config (table-parity), контракт `source_kinds` (реализован для order/sub_order через `Andi.Warehouse.Integration`).

Ограничение: `Stock.location_uuid` — мягкий `Ecto.UUID` без FK, «the app is single-warehouse for now»; к phoenix_kit_locations не привязан.

## 1. Мульти-склад (приоритет 1)

Цеховик: несколько складов (производства, готовой продукции, снабжения), остатки «По товарам / По складам».

- Привязать `Stock.location_uuid` (и location-поля документов) к реальным `phoenix_kit_locations` типа **Warehouse** — резолв имени через API locations, пикер из locations (план locations §5);
- селектор склада в документах приёмки/расхода/инвентаризации (default warehouse — настройка, как сейчас);
- остатки: группировка/фильтр по складу («По складам»), сводная по всем складам;
- у каждого склада — флаги поведения (Цеховик: «что можно принимать/выдавать с этого склада»): склад материалов / готовой продукции / универсальный. Хранить в `features`/`data` Location или в настройках склада на стороне warehouse — решить при реализации.

## 2. Перемещения (Transfers) — новый тип документа

Цеховик: «Перемещения» — стандартная операция между складами.

- Документ: склад-источник, склад-приёмник, строки (позиция, количество), статусы draft → in_transit → done;
- проводки: минус на источнике, плюс на приёмнике (через stock_ledger, атомарно);
- та же обвязка, что у существующих документов: column_config, activity log, comments, файлы.

## 3. Адресное хранение (фаза 2, после locations v0.4)

- `space_uuid` на строках остатков и документов — адрес внутри склада (зона/ряд/стеллаж из дерева Spaces);
- отображение полного пути места (`full_path`);
- инвентаризация по зоне (фильтр строк инвентаризации по поддереву Space);
- потом: подсказка мест при приёмке (put-away), QR-коды мест.

## 4. Связка с производством (вместе с manufacturing v0.4)

Цеховик: материалы списываются в производство, готовая продукция через «СГПП» попадает в приёмку склада ГП.

- Регистрация source kind `production_order` в конфиге andi (по образцу order/sub_order) — изменений в warehouse почти не требует;
- Goods Issue «в производство» — строки из входных строк производственного заказа (`build_lines`);
- Goods Receipt готовой продукции — из выходных строк; очередь «готово к приёмке» (аналог СГПП) — представление на стороне производства, склад просто принимает;
- committed_quantities: резерв материалов под запущенные производственные заказы → видна нехватка.

## 5. Контроль дефицита (Цеховик: «Контроль нехватки деталей»)

- Мин. остаток (reorder point) на позицию (или позицию+склад);
- отчёт/виджет «дефицит»: остаток − резерв < минимума, с учётом потребности открытых производственных заказов;
- прямой переход «создать заказ поставщику» из строки дефицита.

## 6. Закупочные цены и автопересчёт (Цеховик: «История закуп. цен», «Автоматическое изменение цен»)

- История закупочных цен по позиции из строк приёмок (номер приёмки, было/стало, дата) — вкладка в карточке позиции;
- средняя/последняя закупочная цена как основа себестоимости;
- хук в catalogue: при изменении закупочной цены — пересчёт продажных цен по правилу наценки (этап 2 markup mapping в catalogue уже отложен — согласовать с этим планом).

## 7. Связанные документы (Цеховик: «Контроль через связанные документы»)

Цепочка: заказ поставщику → счёт → приёмка → (перемещение) → расход в производство → приёмка ГП → отгрузка.

- У документов уже есть source_refs/ссылки — добавить единое представление «граф связанных документов» в карточке документа (карточки-статусы, как в Цеховике);
- MVP: вертикальный список связей вверх/вниз по цепочке; визуальный граф — потом.

## 8. Отчёты

- **Обороты** (Цеховик: «Обороты»): приход/расход/сальдо за период по позиции/складу;
- экспорт XLS/PDF по образцу существующих таблиц;
- виджеты dashboards: остатки ниже минимума, приёмки за неделю, зависшие перемещения.

## 9. Порядок этапов

1. Мульти-склад (§1) — фундамент, тянет за собой locations v0.3/v0.5.
2. Перемещения (§2) — сразу после мульти-склада.
3. Производственная связка (§4) — синхронно с manufacturing v0.3–v0.4.
4. Контроль дефицита (§5) + обороты (§8).
5. Закупочные цены (§6), связанные документы (§7).
6. Адресное хранение (§3) — после locations v0.4 (UI дерева).

## 10-а. Решения, принятые 2026-07-10 (обсуждение с владельцем)

Объём первой очереди утверждён, план скорректирован:

1. **Берём сейчас**: мульти-склад (§1) → перемещения (§2) → контроль дефицита (§5, полный вариант) + обороты (§8) → связанные документы (§7, список-MVP).
2. **Мульти-склад БЕЗ «флагов поведения»** — все склады универсальные (селектор склада в документах + остатки по складам). Специализация (склад ГП и т.п.) — когда появится производственная связка и реальная потребность разделять потоки.
3. **Дефицит — полный вариант сразу**: «доступно = остаток − резерв» против минимального остатка + переход «создать заказ поставщику». Резервы уже считаются (committed_quantities по заказам/подзаказам); производственная потребность добавится позже сама.
4. **§6 (закупочные цены) СНЯТ со склада** — история закупочных цен относится к карточке позиции каталога, не к складскому модулю. В складе не реализуем; возможное будущее — тема каталога.
5. **Связанные документы — список-MVP** в карточке документа (вверх/вниз по цепочке, статусы, переходы); граф-визуализация — позже при потребности.
6. **Отложено**: адресное хранение (§3, после locations v0.4 и реальной потребности), производственная связка (§4, manufacturing на паузе — ждём обновлений upstream-модуля), виджеты dashboards.

## 10. Взаимосвязи

- **Locations**: склады = Location типа Warehouse; адреса хранения = Spaces (zone/aisle/shelf); пикер и full_path — из плана locations §5.
- **Manufacturing**: source kind `production_order`; резервы материалов; приёмка готовой продукции.
- **Catalogue**: закупочные цены/наценки; позиции строк документов.
